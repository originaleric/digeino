# 2026-04-24 前后端渲染映射规则治理方案

## 目标

在不破坏现有 `Parse -> []Block -> HTML` 能力的前提下，给 `[]Block -> 前端卡片 JSON` 增加“可配置、可版本化、可跨端复用”的规则治理能力，解决规则只存在前端代码里导致的维护分裂问题。

## 现状判断

- `pkg/render` 已经把“文本识别规则”放在 `parse`，把“HTML 呈现规则”放在 `render`。
- 当前 React 前端是“结构化 JSON -> 卡片组件”渲染，核心问题是：`BlockKind` 与卡片 DSL 的映射缺少统一来源。

## 备选路线

- **路线 A（前端约定）**：后端只回 `[]Block`，前端本地映射到卡片。
  - 优点：实现快、后端无侵入。
  - 缺点：多前端/多版本时规则漂移，难审计。
- **路线 B（后端定义）**：后端持有映射规则并直接下发卡片 JSON。
  - 优点：规则集中治理、可灰度、可审计。
  - 缺点：前端灵活度下降，协议演进成本上升。
- **路线 C（推荐，混合）**：后端定义“规则元数据 + 可选已映射结果”，前端保留最后一跳渲染。
  - 做法：
    - 后端返回 `blocks`（真源数据）
    - 可选返回 `ui_model`（按规则映射后的卡片 JSON）
    - 始终返回 `mapping_version`、`mapping_source`（config/db/git tag）
  - 价值：兼顾治理与前端自治，支持 A/B 与回滚。

## 推荐落地（开源友好）

1. 在 `pkg/render` 之外新增一个轻量子包（例如 `pkg/render/uimodel`），职责仅为 `[]Block -> UIModel`。
2. 规则采用 YAML（与现有 `parse`/`render` 同风格），例如：
   - `block_profiles.thinking -> card: ReasoningCard`
   - `block_profiles.code -> card: CodeCard`
   - `fields`, `props`, `actions` 可配置
3. 后端 API 约定三档输出：
   - `mode=blocks`（仅块）
   - `mode=ui_model`（仅卡片 JSON）
   - `mode=both`（推荐默认，便于前端兜底）
4. 加入版本治理字段：`mapping_version`, `mapping_hash`, `mapping_changed_at`。
5. 前端渲染策略：优先 `ui_model`，失败回退 `blocks` 本地映射，确保兼容老接口。

## 关键协议草案

- 响应中增加：
  - `blocks: Block[]`
  - `ui_model?: { schema_version, cards[] }`
  - `mapping_version?: string`
- 流式建议：
  - 增量继续发原始 delta / blocks
  - 在 `done` 事件一次性发 `ui_model`（与现有 done 附加 html 的思路一致）

## 风险与控制

- 规则复杂度膨胀：先限制 DSL 范围（仅 `card_type + props`），暂不做表达式引擎。
- 多端不一致：以 `mapping_version` 为准做前后端兼容矩阵。
- 安全：`ui_model` 不允许任意 HTML 透传，代码/markdown 走白名单字段。

## 首批实施边界（建议）

- 只支持三类块：`markdown/thinking/code`
- 只定义 3 张卡片：`MarkdownCard/ReasoningCard/CodeCard`
- 不做 DB 动态规则，先文件配置 + 热重载（或重启加载）

## 与当前项目的对应

- DigEino 侧：保持 `pkg/render` 聚焦 parse/render（文本与 HTML）
- DigStockAgent（或业务后端）侧：新增 `uimodel` 映射层与 API 输出开关
- React 前端：从“硬编码映射”迁到“优先消费后端映射，保留本地回退”

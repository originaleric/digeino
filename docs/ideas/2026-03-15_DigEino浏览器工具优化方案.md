# DigEino 浏览器工具优化方案

基于对 [pinch-tab](https://github.com/pinchtab/pinchtab) 的分析，本文档提出 DigEino 本地浏览器爬取工具的优化方案。

## 一、核心差异分析

### 1.1 架构对比

| 特性           | DigEino (当前) | pinch-tab                             | 优化方向                     |
| -------------- | -------------- | ------------------------------------- | ---------------------------- |
| **浏览器库**   | go-rod         | chromedp                              | 考虑引入 chromedp 或混合使用 |
| **元素引用**   | CSS 选择器     | 无障碍树引用 (e0, e1, e2)             | **引入无障碍树引用系统**     |
| **快照机制**   | 无             | 结构化快照（可交互元素）              | **新增快照功能**             |
| **标签页管理** | 页面级会话池   | 标签页隔离 + TabExecutor              | **改进并发控制**             |
| **操作类型**   | 基础操作       | 15+ 种操作（含 humanClick/humanType） | **扩展操作能力**             |
| **实例管理**   | 单浏览器实例   | 多实例 + Profile 管理                 | **支持多实例**               |
| **Token 效率** | 全文提取       | 结构化提取（~800 tokens/page）        | **优化输出格式**             |

### 1.2 关键技术点

**pinch-tab 的核心优势：**

1. **无障碍树（Accessibility Tree）引用**：基于 `Accessibility.getFullAXTree` 生成稳定元素引用
2. **快照系统**：提取可交互元素的结构化描述，而非完整 HTML
3. **TabExecutor**：标签页级并发控制，单标签页顺序执行，多标签页并行
4. **操作注册表**：可扩展的操作系统，支持自定义操作

## 二、优化方案

### 2.1 引入无障碍树引用系统（高优先级）

**目标**：提供稳定的元素引用机制，避免 CSS 选择器因页面变化而失效。

**实现路径**：

- 在 `tools/research/` 下新增 `accessibility.go`
- 实现 `FetchAXTree()` 函数，调用 Chrome DevTools Protocol 的 `Accessibility.getFullAXTree`
- 生成元素引用映射（如 `e0`, `e1`, `e2`）
- 修改 `browser_browse.go`，支持通过引用操作元素

**参考实现**：

- `pinchtab-main/internal/bridge/snapshot.go` 的 `FetchAXTree()` 函数
- 使用 `chromedp` 库直接调用 CDP API

**影响文件**：

- `tools/research/browser_browse.go` - 添加引用支持
- `tools/research/accessibility.go` - 新增文件

### 2.2 新增快照（Snapshot）功能（高优先级）

**目标**：提供页面结构化快照，提取可交互元素信息，降低 Token 消耗。

**功能设计**：

- 新增工具 `browser_snapshot`
- 返回可交互元素列表（按钮、链接、输入框等）
- 每个元素包含：引用ID、角色、名称、值、是否禁用等
- 支持过滤选项（仅交互元素、仅可见元素等）

**输出格式**：

```json
{
  "url": "https://example.com",
  "title": "Page Title",
  "elements": [
    { "ref": "e0", "role": "button", "name": "Submit", "disabled": false },
    { "ref": "e1", "role": "textbox", "name": "Email", "value": "" },
    { "ref": "e2", "role": "link", "name": "Privacy Policy" }
  ]
}
```

**实现路径**：

- 新增 `tools/research/browser_snapshot.go`
- 复用无障碍树数据，转换为快照格式
- 注册到 `tools/tools.go`

**参考实现**：

- `pinchtab-main/internal/bridge/snapshot.go` 的 `FetchAXTree()` 和快照格式化逻辑

### 2.3 扩展操作能力（中优先级）

**目标**：支持更多浏览器操作类型，提升自动化能力。

**新增操作**：

1. **humanClick / humanType**：模拟人类操作（随机延迟、轨迹）
2. **hover**：鼠标悬停
3. **scroll**：滚动到元素或指定位置
4. **drag**：拖拽操作
5. **select**：下拉框选择
6. **focus**：聚焦元素

**实现路径**：

- 在 `tools/research/browser_browse.go` 中扩展 `BrowserBrowseRequest.Action`
- 或新增独立工具 `browser_action`，专门处理交互操作
- 支持通过元素引用或 CSS 选择器操作

**参考实现**：

- `pinchtab-main/internal/bridge/actions.go` 的操作注册表
- `pinchtab-main/internal/bridge/cdp.go` 的 CDP 操作封装

### 2.4 改进并发控制机制（中优先级）

**目标**：优化浏览器会话管理，支持标签页级隔离。

**当前问题**：

- `browser_pool.go` 使用页面级会话池，每次操作创建新页面
- 缺乏标签页生命周期管理
- 并发控制较简单（仅槽位限制）

**优化方案**：

- 引入类似 `TabExecutor` 的机制
- 支持标签页复用（同一标签页多次操作）
- 标签页级锁，确保单标签页顺序执行
- 全局并发限制（多标签页并行）

**实现路径**：

- 重构 `browser_pool.go`，引入标签页概念
- 新增 `tab_manager.go` 管理标签页生命周期
- 修改 `browser_browse.go`，支持标签页 ID 参数

**参考实现**：

- `pinchtab-main/internal/bridge/tab_executor.go` 的并发控制逻辑
- `pinchtab-main/internal/bridge/tab_manager.go` 的标签页管理

### 2.5 支持多实例和 Profile 管理（低优先级）

**目标**：支持多个浏览器实例并行运行，每个实例独立的 Profile（Cookie、历史等）。

**应用场景**：

- 不同用户账号的并行操作
- 测试环境的隔离
- 分布式爬取任务

**实现路径**：

- 新增 `browser_instance.go` 管理浏览器实例
- 支持实例创建、销毁、列表查询
- Profile 目录隔离（每个实例独立用户数据目录）
- 修改 `browser_browse.go`，支持指定实例 ID

**参考实现**：

- `pinchtab-main/internal/orchestrator/` 的实例管理
- `pinchtab-main/internal/bridge/init.go` 的 Chrome 初始化

### 2.6 优化 Token 效率（中优先级）

**目标**：减少输出 Token 数量，提高大模型处理效率。

**当前问题**：

- `browser_browse` 返回完整 Markdown，可能包含大量无关内容
- 缺乏结构化提取选项

**优化方案**：

1. **快照模式**：仅返回可交互元素（~800 tokens）
2. **选择性提取**：支持指定选择器范围
3. **摘要模式**：返回页面摘要而非全文

**实现路径**：

- 在 `BrowserBrowseRequest` 中新增 `mode` 参数（`full`/`snapshot`/`summary`）
- 修改 `extractMainContent()`，支持模式切换

### 2.7 考虑引入 chromedp（可选）

**目标**：评估是否引入 `chromedp` 作为 `go-rod` 的补充或替代。

**优势**：

- 更底层的 CDP 控制
- 更好的无障碍树支持
- 与 pinch-tab 技术栈一致

**劣势**：

- 需要重构现有代码
- 学习成本
- 可能与 go-rod 功能重叠

**建议**：

- 第一阶段：保持 go-rod，通过 CDP 调用补充功能
- 第二阶段：评估混合使用或迁移的可行性

## 三、实施优先级

### 第一阶段（核心功能）

1. ✅ **引入无障碍树引用系统** - 提升稳定性
2. ✅ **新增快照功能** - 降低 Token 消耗
3. ✅ **扩展操作能力** - 提升自动化能力

### 第二阶段（架构优化）

1. ✅ **改进并发控制** - 提升性能和稳定性
2. ✅ **优化 Token 效率** - 提升处理效率

### 第三阶段（高级特性）

1. ⚠️ **支持多实例和 Profile** - 按需实现
2. ⚠️ **考虑引入 chromedp** - 长期评估

## 五、兼容性考虑

### 5.1 向后兼容

- 保持现有 `browser_browse` 工具 API 不变
- 新功能通过新工具或可选参数提供
- 现有配置项继续有效

### 5.2 依赖管理

- 如需使用 `chromedp`，评估依赖大小和兼容性
- 优先使用 go-rod 的 CDP 能力（如果支持）

## 六、测试建议

1. **单元测试**：无障碍树解析、快照生成
2. **集成测试**：完整操作流程（导航→快照→操作）
3. **性能测试**：并发场景下的稳定性
4. **兼容性测试**：不同网站的反爬策略

# 2026-04-25 digeino 模板渲染层上提方案

## 背景与问题

- 当前项目为实现期权策略师 HTML 卡片，需要在业务侧维护结构生成代码（如模板渲染服务或字符串拼接逻辑）。
- 这会导致多个项目重复实现同类能力：模板加载、缓存、函数注入、转义安全、错误处理。
- 目标是把“结构生成层”从业务项目上提到 `digeino`，让业务项目只提供模板和数据。

## 目标

- 在 `digeino` 提供通用的 **数据 -> HTML 片段** 模板渲染能力。
- 业务项目仅负责：
  - 提供模板文件（如 `config/templates/*.tmpl`）
  - 提供样式配置（如 `render.yaml` / 独立 css）
  - 组织视图数据（view model）
- 保持安全默认（自动 HTML 转义），避免把渲染责任散落在项目代码中。

## 设计原则

- 安全优先：默认采用 `html/template`，禁止默认关闭转义。
- 可配置优先：模板路径、函数集、缓存策略可配置。
- 项目最小侵入：项目保留数据准备逻辑，不再维护 HTML 结构拼装实现。
- 与现有 render 能力并存：不破坏 `Parse -> Markdown -> HTML` 路径。

## 能力边界

- `render.yaml` 适合配置样式与策略，不负责从 DTO 自动生成复杂结构。
- 模板渲染层负责结构生成，但不承担业务计算（如风控规则推导）。
- 业务项目仍需做轻量 view model 预处理（格式化、空值兜底、字段对齐）。

## digeino 侧能力提案

### 1) 新增模板渲染 API

- `RenderTemplate(templateText string, data any, opts TemplateRenderOptions) (string, error)`
- `RenderTemplateFromFile(path string, data any, opts TemplateRenderOptions) (string, error)`

`TemplateRenderOptions` 建议包含：

- `Funcs map[string]any`：可注册安全函数（如数值格式化、标签映射）
- `Cache bool`：是否启用模板缓存
- `CacheKey string`：显式缓存键（可选）
- `BaseDir string`：模板根目录（相对路径解析）
- `Strict bool`：模板缺字段时是否报错

### 2) 缓存策略

- 默认按 `absPath + mtime` 缓存编译后的模板。
- 本地开发可关闭缓存或使用短周期刷新。
- 生产环境默认开启缓存，降低模板解析开销。

### 3) 错误模型

- 模板文件不存在
- 模板语法错误
- 模板执行错误（字段缺失、函数报错）
- 非法函数注册

建议返回可区分错误类型，便于业务降级处理。

## 项目侧接入方式（以 DigStockAgent 为例）

### 1) 项目保留

- 模板文件：`config/templates/strategist_card.tmpl`
- 样式配置：`config/render.yaml` 中卡片样式（或独立 css）
- 数据准备：`StrategistResponse` -> `TemplateViewModel`

### 2) 项目移除

- 业务侧模板加载缓存实现（可下沉后删除）
- 业务侧 HTML 结构拼装细节

### 3) 调用链建议

- `controller` 中在 `include_strategist_render_html=true` 时：
  - 组织 `view model`
  - 调 `digeino.RenderTemplateFromFile(...)`
  - 将片段包裹样式容器后返回 `strategist_html`

## 迁移步骤

1. 在 `digeino` 增加模板渲染能力及单元测试。
2. 在 DigStockAgent 引入新 API，替换本地模板渲染实现。
3. 保留原链路一段灰度期（可通过配置切换）。
4. 验证稳定后删除项目内重复实现代码。

## 验收标准

- 功能：策略师 `strategist_html` 渲染与当前模板输出一致。
- 性能：高并发下模板渲染耗时与 CPU 占用在可接受范围。
- 安全：动态文本默认转义，禁止未受控 raw HTML 注入。
- 可维护性：新增页面仅需模板 + 数据映射，无需再写结构拼接代码。

## 风险与回退

- 风险：上提后若 API 设计不稳，可能影响多个项目接入节奏。
- 缓解：先发布 `experimental` 接口，稳定后转正式 API。
- 回退：项目保留本地模板渲染实现开关，必要时快速切回。

## 后续建议

- 与 UIModel 方向结合：模板渲染先解决“结构可配置”，UIModel 再解决“跨端结构统一”。
- 统一模板函数库：数字、货币、百分比、风险标签映射在 digeino 侧沉淀为通用 helper。

## React/Vue 技术栈兼容策略（补充）

### 核心判断

- 目标不是让 Go 端直接运行 React/Vue 运行时，而是让 React/Vue 团队可低成本接入同一套渲染协议。
- 不建议把完整 TSX/Vue SFC 直接编译到 Go 模板：语义差异大、维护成本高、边界复杂。

### 推荐路线 A（优先）：UIModel-first

- `digeino` 输出结构化 `UIModel JSON`（如 card/list/table/grid/meta）。
- 前端按技术栈实现渲染器：
  - React: `UIModel -> React Components`
  - Vue: `UIModel -> Vue Components`
- 优点：
  - 跨栈一致性最好
  - 可测试性高（结构回归测试）
  - 无模板语法绑定风险

### 推荐路线 B（次选）：中立模板引擎

- 在 `digeino` 的模板抽象层中支持非 Go 专属模板（如 Mustache / Handlebars / Liquid）。
- 保留 `go html/template` 作为默认引擎，同时允许按项目选择模板引擎。
- 优点：
  - React/Vue 团队可使用更熟悉的模板语法
  - 迁移成本较低
- 限制：
  - 仍是 HTML 片段输出，不等价于组件运行时能力

### 不建议作为一期目标

- 直接支持“完整 TSX/Vue 模板 -> Go 渲染”。
- 原因：
  - 需要引入 AST 解析、语义降级、运行时特性裁剪
  - 复杂度接近编译器工程，不利于快速稳定落地

### 分阶段建议

1. 一期：完成 `digeino` 模板抽象层 + 多模板引擎扩展点（保持安全默认）。
2. 二期：提供 `UIModel schema` 与 React/Vue 参考渲染器。
3. 三期（可选）：评估“受限 TSX/Vue 子集”的转换工具，仅覆盖少量静态模板场景。

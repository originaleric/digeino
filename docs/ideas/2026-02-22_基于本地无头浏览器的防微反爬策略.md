# Idea: 基于本地无头浏览器的高级反爬抓取方案 (2026-03-08 更新版)

## 1. 背景与痛点

在抓取微信公众号、知乎、极客时间等平台时，传统的 HTTP 爬虫（Firecrawl, Jina Reader）受限于公有云 IP 质量和固定指纹，极易遭遇"环境异常"、"验证码"或"Forbidden"拦截。为了实现 100% 的抓取成功率，引入本地 Headless 浏览器驱动是目前公认的终极方案。

## 2. 核心推荐 Go 爬虫库对比

| 库名称 | 特点 | 适用场景 |
| :--- | :--- | :--- |
| **[go-rod](https://github.com/go-rod/rod)** | **首选推荐**。API 人性化，社区活跃。其 `stealth` 插件极其强大。 | 对抗微信、Cloudflare 等具有深度 JS 指纹检测的平台。 |
| **[chromedp](https://github.com/chromedp/chromedp)** | 原生 Go 实现，轻量级。直接操作 CDP 协议。 | 高并发、需要精细化控制浏览器行为的底层开发。 |
| **[playwright-go](https://github.com/playwright-community/playwright-go)** | Microsoft 官方协议的 Go 社区分支，稳定性极高。 | 需要跨浏览器（Firefox/WebKit）兼容性的复杂自动化。 |

*注：经过多轮方案论证，我们最终放弃了 `chromedp`，转向采用 `go-rod` + `stealth` 插件，因为它在防微反爬、抹除特征方面的能力最强，并且自带 launcher 减轻运维负担。*

## 3. 为什么本地实现能绕过高级反爬？

- **真实环境模拟**: 运行完整的 Chromium 引擎，执行所有 JavaScript 脚本，避开"仅检测 HTTP Header"的初级拦截。
- **隐藏自动化特征 (Stealth)**: 通过插件抹除 `navigator.webdriver` 等标识，使浏览器在特征检测中表现得像真实用户。
- **IP 信誉度**: 本地开发或家庭宽带 IP 的信誉权重远高于 AWS/阿里云等数据中心 IP。
- **Cookie 与场景持久化**: 能够持久化存储访问历史，模拟自然人的连续访问行为。

## 4. 架构建议 (Smart Fallback Strategy)

在 `DigEino` 调研工具集中采用"阶梯式切换"逻辑，将浏览器能力封装为通用插件工具 `browser_browse` 供 Agent（如 DigFlow）调用：

```mermaid
graph TD
    Agent["Web Researcher Agent"] --> Router{"抓取策略路由"}
    
    Router --> |首先尝试 (L1)| L1["L1: firecrawl / jina (云端HTTP)"]
    L1 --> Check{"是否遭遇拦截？<br>如验证码、环境异常"}
    
    Check -->|否, 成功| Success["提取 Markdown 返回给 LLM"]
    Check -->|是, 被拦截| L2["L2: browser_browse<br>(本地 go-rod + stealth 浏览器)"]
    
    subgraph CentOS Server (应用宿主机)
        L2 --> ChromePool["go-rod 浏览器实例池 (限制并发)"]
        ChromePool --> HeadlessChrome["无头 Chromium 进程 (带 stealth 插件)"]
    end
    
    HeadlessChrome -.-> |绕过 JS 指纹检测| TargetWeb["微信公众号 / Cloudflare站点"]
    TargetWeb -.-> |渲染后的 DOM| HeadlessChrome
    HeadlessChrome --> DOM2MD["html-to-markdown"]
    DOM2MD --> Success
    
    L2 --> L2Check{"是否超时/崩溃？"}
    L2Check --> |是| L3["L3: web_search<br>仅获取搜索引擎的快照摘要"]
    L3 --> Success
```

## 5. 工具功能设计 (browser_browse Tool)

为了让大模型 (LLM) 能够智能调用，`browser_browse` 提供以下可控参数：

**工具输入参数 (Input JSON)：**

- `url` (string, 必填)：需要访问的目标网页。
- `action` (string, 选填)：动作类型。
  - `"read"`：提取正文（转为 Markdown）。
  - `"screenshot"`：截取网页全屏（返回 Base64，备用多模态场景）。
- `wait_selector` (string, 选填)：CSS选择器，等待特定元素加载（解决 JS 异步渲染慢）。
- `use_cookie_domain` (string, 选填)：指定加载特定域名的预设 Cookie（突破登录墙，如带上企业内部 Wiki 的 Cookie）。

**工具处理流程：**

1. 从资源池获取一个复用的 Chrome Tab。
2. 注入 `stealth` 脚本防止被识别为爬虫。
3. 访问 URL，等待网络空闲或 `wait_selector` 出现。
4. 剥离广告/侧边栏，将 DOM 核心正文使用 `html-to-markdown` 转为纯文本。
5. 返回结果并回收 Tab。

## 6. 部署与环境说明

### 开发环境 (macOS)

- **零配置体验**: 在 Mac 上测试时，`go-rod` 会自动拉起一个独立的 Chromium 实例。
- **可视化调试**: 可将 `Headless(false)` 设为可见模式，实时看机器人如何滚动页面。

### 生产环境 (CentOS 服务器准备)

在不改动代码前，CentOS 需要做如下准备：

1. **安装无头浏览器基础库**：
   `go-rod` 虽会自动下载 Chromium，但 CentOS 运行它需要依赖这些底层图形库：

   ```bash
   sudo yum install -y libX11 libXcomposite atk gtk3 cups-libs
   ```

2. **安装中文字体（非常关键）**：
   如果不装字体，截图或者渲染时中文会变成乱码小方块：

   ```bash
   sudo yum install -y google-noto-cjk-fonts
   # 或 sudo yum install -y wqy-microhei-fonts wqy-zenhei-fonts
   ```

3. **并发与内存预估（风险控制）**：
   - 官方经验：每个并发 Chromium 会话约占 100MB~300MB 内存。
   - DigFlow 主服务服务器（如 4核8G）上运行，建议连接池的 `CONCURRENT` 限制设置为 3-5，预留最多 1-2GB 内存给爬虫专用，防止把主服务拖垮。

4. **沙盒权限配置**：
   由于 CentOS 默认限制，代码里需配置 `--no-sandbox` 和 `--disable-gpu`。

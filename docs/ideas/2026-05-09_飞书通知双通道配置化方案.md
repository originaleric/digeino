# 2026-05-09 飞书通知方案（完全参考 OpenClaw）

## 一、目标

在 DigEino 中新增飞书通知能力，完全对齐 OpenClaw 的主路径设计：

- 入站接入：默认 WebSocket（可选 webhook）
- 出站发送：统一 HTTP API（单路径）

本期不引入双通道并存、广播、去重等额外复杂机制。

---

## 二、对齐原则

OpenClaw 的关键模式是：

1. 飞书事件接入默认走 WebSocket（不依赖公网 webhook）
2. webhook 作为可切换接入模式
3. 消息发送本质走飞书开放平台 HTTP API

因此本方案采用：

- `ConnectionMode=websocket` 作为默认值
- 出站仅走 `HTTP API`
- webhook 仅用于“入站事件接收模式切换”，不是并行发送通道

---

## 三、为什么不做双通道

- OpenClaw 主路径是单路模型，不需要额外路由策略
- 单路径配置更清晰，减少误配与排障成本
- 对当前目标“快速落地飞书通知”更稳妥

---

## 四、配置设计（核心）

建议新增 `Feishu` 配置段，示例：

```yaml
Feishu:
  Enabled: true

  # 接入层（参考 OpenClaw，默认 websocket）
  ConnectionMode: "websocket"     # websocket | webhook
  EventIngest:
    WebsocketEnabled: true
    WebhookEnabled: false
    VerificationToken: ""         # webhook 模式需要
    WebhookPath: "/feishu/events"
    WebhookHost: "127.0.0.1"
    WebhookPort: 3000

  # 发送层（单路径 API）
  SendViaAPI: true

  NotifyOnEvents: ["started", "completed", "failed"]  # 自动通知事件

  API:
    Enabled: true
    AppID: ""
    AppSecret: ""
    BaseURL: "https://open.feishu.cn"
    Timeout: 5
    RetryCount: 2
    RetryDelay: 500
```

说明：

- `ConnectionMode` 决定入站事件接入方式（默认 websocket）
- `SendViaAPI=true` 表示出站统一走 API
- 不提供双通道并发与广播配置，保持与 OpenClaw 主路径一致

---

## 五、发送策略建议

### 5.1 默认策略（推荐）

- `ConnectionMode=websocket`
- `SendViaAPI=true`

行为：消息统一通过 API 发送，不做双通道路由决策。

### 5.2 webhook 模式说明

当 `ConnectionMode=webhook` 时，仅影响“飞书事件如何进入 DigEino”，不改变出站 API 发送方式。

---

## 六、能力边界与触发方式

需要同时支持两种触发方式：

1. 自动触发：状态事件（started/completed/failed）由配置驱动发送
2. 手动触发：提供 `send_feishu_message` 工具供 Agent 主动调用

这样可以覆盖“流水通知”与“人工补发/定向通知”两类需求。

## 七、与现有架构的衔接建议

- 复用 `webhook/status_collector.go` 的异步分发机制
- 在 `tools` 体系新增飞书发送工具并按 `Enabled` 决定是否注册
- API 客户端按现有配置/重试风格实现
- 配置默认值保持关闭，保证向后兼容

---

## 八、验收标准

1. `ConnectionMode=websocket` 为默认，并可切换 `webhook`
2. 自动通知与手动工具调用均可成功发送
3. 出站消息统一 API 发送并稳定可观测
4. 不影响现有微信、企业微信与通用 webhook 行为

---

## 九、落地优先级

1. 先实现配置结构：`ConnectionMode` + `SendViaAPI` + API 凭证
2. 实现出站 API 客户端与自动/手动发送能力
3. 实现入站 websocket（默认）与 webhook（可切换）
4. 补齐测试与文档

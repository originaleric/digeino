# 2026-03-28 DigEino执行事件底座升级与使用说明

## 概述

本次升级围绕 `DigEino` 事件底座完成了 P0/P1/P2 三阶段核心能力落地，目标是：

- 统一事件语义（生命周期标准化）
- 保持历史字段兼容（零中断升级）
- 提升分发稳定性（异步队列、重试、降级）
- 提供运行策略控制（采样、payload 限制）
- 提供运行可观测性（分发统计）

本次升级仅在 `DigEino` 范围内，不涉及 DigFlow 治理层能力。

## 本次新增能力

### 1) 统一事件模型（兼容历史字段）

在 `ExecutionStatus` 上新增并行字段：

- `schema_version`（当前 `1.0`）
- `event_type`（`started/succeeded/failed/retried/skipped/completed`）
- `node_id`（与 `node_key` 并行）
- `attempt`
- `source`

同时保留历史字段：

- `type`（`node_start/node_end/complete`）
- `node_key`

并提供：

- `NormalizeEventType()`：统一新旧事件语义
- `AsExecutionEvent()`：转换到统一事件结构
- `DedupeKey()`：推荐去重键（`execution_id|node_id|attempt|event_type`）

### 2) Sink 分发增强（稳定性）

`StatusCollector` 的 store/webhook 分发从“直接 goroutine”升级为“队列分发模型”：

- 内部异步队列（削峰）
- 分发重试（默认 3 次）
- 队列拥堵降级（超时后独立 goroutine 兜底）
- 主回调（callback）仍优先执行，保持实时体验

### 3) 统一入口初始化

新增 `webhook.NewConfiguredCollector(...)`，用于统一 HTTP/CLI/批处理三类入口的 collector 装配逻辑，避免每个入口手写装配导致行为不一致。

### 4) 事件策略与可观测

新增策略能力：

- `Status.Event.SampleRate`：非终态事件采样率（0-100）
- `Status.Event.MaxPayloadBytes`：单事件 payload 上限

新增运行统计能力：

- `GetDispatchStats()` 返回分发统计（入队、重试、失败、采样丢弃、压缩次数等）

## 配置说明

在 `config/config.yaml` 中增加如下配置：

```yaml
Status:
  Webhook:
    Enabled: false
    URL: ""
    Events: ["complete", "error"] # 兼容旧名；也支持 started/succeeded/failed/completed
  Store:
    Enabled: true
    Type: "memory"
  Event:
    SampleRate: 100            # 非终态事件采样率（0-100），100 表示不采样
    MaxPayloadBytes: 262144    # 单条状态事件 payload 上限（字节）
```

说明：

- `SampleRate` 只作用于非终态事件；终态事件（`completed`/`failed`）始终发送。
- `MaxPayloadBytes` 超限时会触发渐进压缩（优先裁剪 DataFlow 明细）。

## 如何使用（详细）

### 步骤 1：初始化配置

```go
_, err := config.Load("config/config.yaml")
if err != nil {
    panic(err)
}
```

### 步骤 2：创建状态存储（可选）

```go
baseStore := status.GetDefaultStore()
store := status.NewStatusStoreAdapter(baseStore)
```

### 步骤 3：统一创建 Collector（推荐）

```go
collector := webhook.NewConfiguredCollector(
    executionID,
    "YourAppName",
    requestID,
    store, // 可为 nil
    func(s webhook.ExecutionStatus) {
        // SSE/WS 推送等实时回调（可选）
    },
    func() string {
        // 仅在未配置 URL 时作为默认值提供（可选）
        return webhook.BuildDefaultWebhookURL("http", "")
    },
)
if collector != nil {
    // 可选：运行时覆盖策略（优先级高于配置）
    collector.SetEventPolicy(100, 262144)
    ctx = webhook.WithStatusCollector(ctx, collector)
}
```

### 步骤 4：在执行流程中上报节点生命周期

```go
sc := webhook.GetStatusCollector(ctx)
if sc != nil {
    sc.OnNodeStart(ctx, "node_1", "chat_model", input)
}

// ... 执行节点逻辑 ...

if sc != nil {
    sc.OnNodeEnd(ctx, "node_1", "chat_model", output, execErr)
}
```

### 步骤 5：执行结束时上报完成事件

```go
sc := webhook.GetStatusCollector(ctx)
if sc != nil {
    sc.OnComplete(ctx, resultMessage, finalErr)
}
```

### 步骤 6：读取分发统计（调试/监控）

```go
if sc != nil {
    stats := sc.GetDispatchStats()
    // stats 包含：Enqueued / RetryCount / StoreFailure / WebhookFailure / SampledOut 等
    _ = stats
}
```

## 事件字段迁移建议

### 推荐消费优先级

1. 优先使用 `event_type`（统一语义）
2. 优先使用 `node_id`（统一节点标识）
3. 使用 `schema_version` 做版本分支
4. 保留对 `type`、`node_key` 的兼容读取

### 旧新字段映射

| 历史字段 | 新字段 | 说明 |
| --- | --- | --- |
| `type=node_start` | `event_type=started` | 节点开始 |
| `type=node_end,status=success` | `event_type=succeeded` | 节点成功结束 |
| `type=node_end,status=error` | `event_type=failed` | 节点失败 |
| `type=complete,status=success` | `event_type=completed` | 执行完成 |
| `type=complete,status=error` | `event_type=failed` | 执行失败 |
| `node_key` | `node_id` | 节点标识并行输出 |

## 兼容性说明

- 现有消费者继续读取 `type/node_key` 不会中断。
- Webhook `Events` 支持新旧事件名混用（如 `["complete","error"]` 或 `["completed","failed"]`）。
- 默认行为不需要强制配置 `Status.Event`，不配置时会使用安全默认值。

## 验证方式

建议按以下顺序验收：

1. 执行 `go test ./...`，确保单测通过。
2. 开启一个内存 store + 本地 webhook mock，验证能收到 `started/succeeded/completed`。
3. 将 `SampleRate` 调低（如 `10`），确认终态事件仍稳定送达。
4. 通过构造大 payload，确认触发 payload 压缩且流程不中断。

## 常见问题

### Q1：我只订阅了 `error`，还能收到失败事件吗？

可以。失败事件会兼容匹配 `error` 与 `failed`。

### Q2：采样会不会把最终完成事件采样掉？

不会。终态事件不参与采样，始终发送。

### Q3：如果 payload 太大，会直接丢事件吗？

不会优先丢。会先做渐进压缩（裁剪明细）后再分发，尽量保障事件可达。
